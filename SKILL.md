# 生成 / 发布文档(工作流)

把 AI 生成内容**发布并归档到飞书**。核心是编排脚本 `scripts/publish-doc.sh`,把"输入→(html?→上传)→建飞书文档→灌内容→出链接→(可选)发消息"串成一条确定性流水线;**不靠记忆临场拼命令**。

> 首次用先跑 `./setup.sh`(检查依赖 + 写 `config.local.env`)。HTML 上传走内置 `scripts/upload-html.sh`。

## 文档大师约束(生成内容的硬规矩,优先于一切)

把自己当**有极致文档审美的专业作者**,不是流水账机器。每篇必须满足:

1. **零草稿痕迹**:正文里**不留**导语式自述(「一份面向工程师的概念地图…」)、不留「预备稿 / 草稿 / 日期署名」、不留「本文将介绍…」这类废话开场。标题即节点标题,正文直接进内容。
2. **有层级,不流水账**:**禁止** `0. 1. 2. … 11.` 这种一长串同级二级标题。要有真实的信息架构——H2 分主干板块,H2 下按需拆 H3、H4 子层级,轻重分明。读者能从目录看出结构,而不是一条直线。
3. **图随文走**:图**插在它讲解的那一节里**(用 `@@IMG` 标记),不许甩末尾、不许堆一起。一图配一处复杂点,保证阅读顺下来时图正好出现在需要它的位置。
4. **不堆枯燥文字**:复杂处优先用图/表/示例代替大段文字(见「配图」)。每节聚焦一件事,长段落拆开。
5. **可读优先**:开头用一两句点题或一张全局图建立心智模型,再展开;相邻概念讲清边界;结尾给速查/避坑这类高密度收口,而非又一段散文。

> 判定:把成稿当作要挂进公司知识库、给同事当权威参考的文档来要求。达不到就重写,别交草稿。

## 四遍 Loop 精修(生成内容的核心节奏,禁止一遍交稿)

一篇好文档不是一次写成的。**每篇至少过四遍 loop**,每遍目标递进;且有两条**贯穿每一遍的不变量**,每遍结束都要自检:

- **审美不变量**:文本审美(措辞精炼、节奏顺、不啰嗦)+ 图片审美(taste 浅色高级、风格统一)+ 内容审美(信息密度够、逻辑通)三者都要在线。
- **格式不变量**:markdown 格式正确——标题层级、表格、列表、callout、`@@IMG` 标记全部无破损。

**Loop 1 · 定初稿**
把内容写下来,搭出主干结构(H2 板块 + 关键子层级雏形);复杂处先用注释标记"这里要图"。重点是**内容全、骨架对**,不纠结字斟句酌。

**Loop 2 · 强化 + 精简 + 定型**
- **强化内容**:每节补到讲透,删空话/重复。
- **精简内容**:长段落拆开,能用图/表/示例代替的文字就替换。
- **强化格式 + 锁定目录**:**确定最终的文章目录与子章节层级**(H2/H3/H4 分明、命名统一、不流水账),这一遍把结构钉死。
- **配图落位**:把标记处的图真正生成(taste 风格 SVG→PNG),`@@IMG` 插到对应章节。

**Loop 3 · 充实案例与用途(让内容不晦涩,最关键的一遍)**
上几遍容易写成"每点一句话"的骨架——读着多,其实每步都太薄、抽象难懂。这一遍给**每个大章节**按这个三段式补到位,贴合人的认知直觉:

> **一句话概括** → **举个简单例子**(具体、不复杂,能落地想象) → **实际使用用途**(在真实场景里这东西用来干嘛)

例:讲"上下文压缩"不能只说"超窗时摘要替换早期内容";要补——*比如一段聊了两小时的对话塞不下了(例子),系统就把前一小时自动总结成几句话替换掉、保留最近的细节(怎么做),于是你能一直聊下去不断片(用途)*。例子要简单到一眼懂,但必须有。读完每节,读者要能"知道它是什么、长什么样、拿来干嘛",而不是只记住一个名词。

**Loop 4 · 整体审校(最后一遍)**
站在读者视角通读全篇,逐项核:整体结构是否顺 → 每张图是否在对的位置、且讲清了它该讲的 → 文本与内容是否符合需求 → 有无草稿痕迹/跑题/前后矛盾 → **每节是否都做到了"概括+例子+用途"、有没有还停留在干巴一句话的**。发现问题回炉,直到达到"能挂公司知识库当权威参考"的水准再发布。

> 各遍侧重:Loop1 求**全**、Loop2 求**精与定型**、Loop3 求**好懂(例子+用途)**、Loop4 求**整体合格**。全过 + 两条不变量在线,才进发布。

## 落点(飞书 wiki，token 来自 config.local.env,不写死)

落点是一棵飞书 wiki 树,两个父节点(`setup.sh` 时填进 `config.local.env`):
- `FEISHU_PARENT_SHARE` —— 分享/学习类的家
- `FEISHU_ROOT_AUTO` —— 日常自动生成的根(下挂 月/日 目录)

```
FEISHU_ROOT_AUTO  (例:「AI 自动生成」)
├── FEISHU_PARENT_SHARE  (例:「AI 助手知识库」)   ← 分享/学习类 → --kind share
└── 2026-06 (月,脚本自动 find-or-create)            ← 日常 → --kind daily
    └── Agent 2026-6-14 (日文件夹)   ← 当天多篇文档落这里
        ├── [06-14 11:18] 文档A
        └── [06-14 18:02] 文档B
```

- **分享/学习类**(概念/教程/方案综述/调研/技术分享):`--kind share`,挂 `FEISHU_PARENT_SHARE`。
- **日常**(工作摘要/digest 类):`--kind daily`,脚本按月 find-or-create「YYYY-MM」→ find-or-create 当日「Agent Y-M-D」**文件夹** → 在其下建文档(标题自动加 `[MM-DD HH:MM]` 前缀便于排序)。
- 已有自动归档服务(如某后端定时 digest)可共用 `FEISHU_ROOT_AUTO` 这同一棵树:只要月/日命名一致(`YYYY-MM` / `Agent Y-M-D`),两边互不打架。
- 每篇都**新建**,不移动旧文档(部分环境 wiki 移动缺 scope)。

## 格式决策(按内容自动定，不必问用户)

| 内容特征 | 怎么落 |
|---|---|
| **默认**:结构化长文(文字+表格+流程图) | 直接 markdown → 飞书文档(`--md`) |
| 需精致排版/可视化/交互/海报/"一个链接发出去看" | 先生成 `.html` → 脚本 `--html` 自动经 upload-html 上传 → 链接置顶进飞书文档 |
| 用户明确「只要本地文件/草稿」 | 落本地 `[描述]-[YYYYMMDD].md`,不发飞书 |

图片产物统一存一个固定目录(自选,如 `~/doc-img/`),命名 `<场景>-<YYYYMMDD-HHMMSS>.png`。

## 标准工作流

1. **四遍 Loop 精修产出 md**(核心,见上「四遍 Loop 精修」)→ Loop1 初稿、Loop2 强化+精简+定结构+配图落位、Loop3 充实例子与用途、Loop4 整体审校;每遍守审美与格式两条不变量。图标记用 `@@IMG:路径|说明@@` 内联在对应章节。
2. **需要 HTML 吗** → 需要精致排版/可视化/海报就同时生成 `.html`(**浅色高级 taste 风格**,见「视觉统一语言」),交给脚本 `--html` 自动上传。
3. **跑编排脚本**(一条命令完成 建文档→灌内容(含内联图)→出链接→可选通知):
   ```bash
   # 分享/学习类
   scripts/publish-doc.sh \
     --kind share --title "大模型核心概念梳理" --md xxx.md [--html xxx.html] [--notify]

   # 日常自动生成(标题默认 Agent Y-M-D)
   scripts/publish-doc.sh --kind daily --md daily.md [--notify]
   ```
   脚本输出 `doc_url=`(飞书文档链接)、必要时 `html_url=`。
4. **把链接给用户**;`--notify` 会把链接 POST 到 `NOTIFY_URL`(config 里配,如本地通知服务),纯文本带裸 URL。

## 视觉统一语言:taste-skill 浅色高级

本 skill 产出的**一切视觉**(配图 SVG/PNG、HTML 页面)统一走 **taste-skill 的浅色高级方向**——暖白底 + 近黑字 + 单一祖母绿 accent + 克制留白 + 清晰字阶。画图前可参考 taste-skill 校准审美;下方配色即该方向的落地值。

**现成资产**:
- HTML 页面 → 直接内联/链接 [`assets/taste-light.css`](assets/taste-light.css)(Geist 字体、暖白底、祖母绿 accent、卡片/表格/callout 全套样式),正文套 `<div class="doc">`,改不动审美。
- 配图 SVG → 用下方「配图」的配色常量,`rsvg-convert -z 2` 转 PNG。两者同一套色,保证图文一致。

## 配图(让文档不枯燥,生成式 md 必做)

原则:**不要大段枯燥文字硬讲**。任何"复杂处"——概念关系、流程、架构、时序、对比、分层——优先**配一张图**。

- **画什么用什么**:
  - 流程/架构/时序/关系/分层 → 手写**浅色高级风格 SVG**,`rsvg-convert -z 2 x.svg -o x.png` 转 PNG(矢量清晰、风格可控、无需联网)。
  - 数据/图表 → matplotlib 出 PNG。
  - 也可用 mermaid(`<whiteboard>` 在飞书原生渲染,见 lark-whiteboard)代替部分流程图。
- **统一配色(浅色高级)**:底 `#ffffff` / 字 `#1b1b18` / 次要 `#565650` / 弱 `#8e8e82` / hairline `#e6e6df` / accent 祖母绿 `#059669`(淡底 `#ecfdf5`);字体 `PingFang SC, Helvetica Neue, sans-serif`。卡片圆角 14、accent 左条、清晰字阶。
- **存放**:PNG 落一个固定图片目录(自选,如 `~/doc-img/`),命名 `<场景>-<YYYYMMDD-HHMMSS>.png`。
- **插进文档**:在 md 里用单独一行标记 `@@IMG:/绝对路径.png|图说明@@`(说明可省),脚本会**自上而下顺序构建**,把图落在该位置(`docs +media-insert` 只能追加末尾,故必须顺序 append 文字段+插图,脚本已处理)。

## HTML 用浅色高级 taste 风格

需要出 HTML 页面时(海报/可视化/精致排版),按 **taste-skill 的浅色高级方向**画:暖白底 + 近黑字 + 祖母绿 accent + Geist 字体 + 柔和阴影 + 克制交互(见 taste-skill)。生成本地 `.html` → 脚本 `--html` 自动上传拿在线链接。

## 脚本参数(`scripts/publish-doc.sh`)

| 参数 | 说明 |
|---|---|
| `--kind` | `share`(默认,→知识库) \| `daily`(→AI 自动生成/当月) |
| `--title` | share 必填;daily 不填默认 `Agent Y-M-D` |
| `--md` | 必填,正文 markdown 文件;内可含 `@@IMG:路径\|说明@@` 内联图标记 |
| `--html` | 可选,本地 html(浅色高级 taste 风格);脚本自动 upload-html 上传并把链接置顶 |
| `--notify` | 可选,完成后把链接 POST 到 `NOTIFY_URL` 发消息 |

## 发链接到飞书消息(规矩)

`--notify` 走纯文本 + 裸 URL(飞书才出文档预览卡);若手动另发飞书消息,同样禁止用 markdown 包链接(`[文字](url)` 不展开预览),直接放裸 URL。

## 注意

- 落点 token / 密钥全在 `config.local.env`(`setup.sh` 生成,`.gitignore` 忽略),脚本里不写死、不上传。
- 发布 = 在你飞书里建真实文档,按需建,别重复刷屏。
- 短推送(一句话提醒类)不必走本 skill,直接发消息即可。
